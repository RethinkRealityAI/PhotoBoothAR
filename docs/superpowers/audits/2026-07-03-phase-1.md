## Edge function

`submit-post` deployed to the live project (version 1, ACTIVE, verify_jwt on —
anon JWT required). Two-step signed-URL upload; enforces event-live, session-id
format, ext/content-type allowlists, tenant-scoped path prefix (blocks forged
`path` cross-tenant injection), 8MB/60MB size caps with best-effort cleanup,
and a 30-posts/hr per-(event,session) quota.

Known accepted nits: quota check→bump is not atomic (a racing burst can
slightly exceed 30/hr — acceptable for abuse control); direct HTTP smoke-test
from this container is blocked by the environment's network policy (CONNECT 403
to *.supabase.co — MCP calls run remotely and are unaffected). Function logic
verified by line review; end-to-end exercise happens on the deployed site.

## Code audit (adversarial pass over the Phase 1 diff)

Logic items — ALL CLEAN:
- No stray `EVENT_ID`/`activeEvent` outside deliberately-gated legacy paths
  (main.tsx bootstrap, App.tsx legacy tree, store initial state re-keyed by
  `setActiveEvent`).
- submitPost client ↔ submit-post function contract matches (init `{path,
  token}` → `uploadToSignedUrl` → finalize `{post}`).
- Realtime channel names + filters are event-suffixed consistently.
- `VITE_EVENT` legacy builds byte-identical (verified by legacy-mode build +
  green deploy preview on the theadetoyis site).
- Guest localStorage migration (`hopegala.*` → `pbar.<eventId>.*`) correctly
  applies ONLY to the three legacy slugs — new events start clean.

Security findings — 3 found, 3 FIXED (migration `005_phase1_audit_fixes` +
submit-post v2, both live):
1. **Cross-tenant catalog injection**: grandfather policy allowed anon to
   insert/update legacy-slug experiences with `is_global=true`, surfacing
   attacker content in every tenant's catalog. Fixed: grandfather WITH CHECK
   now requires `not is_global`. Live probes: legacy insert without is_global
   allowed ✓; with is_global denied ✓; UPDATE flip denied ✓.
2. **Legacy-org claim on unconfirmed signup**: the signup trigger claimed the
   Legacy org at auth.users INSERT — before email verification, so a stranger
   registering the owner's address would consume the claim. Fixed: claim moved
   to email-confirmation (`on_auth_user_confirmed` trigger + `claim_legacy_org`,
   EXECUTE revoked from clients), defensive cleanup of premature claims. Live
   probe: org unclaimed (owner null, 0 members) ✓.
3. **submit-post init spam**: quota counted only finalized posts, so `init`
   could mint unlimited signed URLs (storage-fill abuse). Fixed: quota now
   consumed at init — deployed as function version 2 (ACTIVE).

Accepted risks (documented): public posts expose `session_id`/`guest_name`
(legacy-compatible `select *`; column-level hardening scheduled with legacy
archive — a knowing attacker could exhaust a specific guest's quota or post
under their session attribution); uploaded bytes are not content-sniffed
(matches legacy behavior); quota check→bump not atomic (burst may slightly
exceed 30/hr); org creation is unthrottled for authenticated users (Supabase
auth rate limits apply upstream).
